# Meetlog フェーズ2 設計書・作業指示書

この文書はClaude Codeへの作業指示書である。プロジェクトの引き継ぎ情報・常時ルールは既存のCLAUDE.md等を参照すること。本書はフェーズ2（Supabase Storage導入・注意書き追加・録音機能）の設計と実装手順を定める。

## 0. 全体方針

- 実装は **ステージA（Supabase Storage＋注意書き）→ ステージB（録音機能）** の2段階で進める。ステージAの動作確認が取れるまでステージBに着手しない。
- ユーザー（ゆう）の手動操作は本書「ユーザー操作チェックポイント」に列挙したものだけに限定する。それ以外の作業はすべてClaude Code側で完結させること。判断に迷う点は実装前にまとめて質問し、細切れに確認を求めない。
- 常時ルールを遵守：作業前に既存コードの問題点を確認し修正と同時にリファクタリング、機能追加前後にgit commit、新機能は🆕付きで報告、節目ごとに実装済み機能一覧を報告。
- UIの注意書き・ガイダンスは画面下部にプレーンテキストで表示。LP化・過剰デザイン禁止。注意書きの文面を新規追加・変更する場合は、実装前に必ずゆうに文面の承認を取ること。
- **STTベンダー方針**：AssemblyAIは「確定した選択」ではなく「差し替え可能なデフォルト」として扱う。フェーズ2でSTT層を抽象化し（A-5参照）、並行タスクのベンダー実測比較（本書末尾）の結果を受けて、フェーズ3でベンダー確定・リアルタイム対応の設計を行う。リアルタイム文字起こし自体はフェーズ2のスコープ外。

## ステージA：Supabase Storage導入 ＋ 注意書き追加

### A-1. アーキテクチャ

現状の「クライアント→Vercel(server)→AssemblyAI」のファイル送信経路を廃し、音声データがVercelを通らない構成に変更する。これによりVercelの4.5MBボディ制限を回避する。

```
[クライアント]
  1. supabase-js(anonキー)でSupabase Storageへ直接アップロード
  2. アップロード先のファイルパスのみをserverへPOST
[server (Vercel)]
  3. service_roleキーで署名付きURL(有効期限2時間)を生成
  4. AssemblyAIへ audio_url として渡し、文字起こしを実行
  5. 文字起こし完了後（成功・失敗問わず）、Storageから当該ファイルを削除
[クライアント]
  6. 結果（話者識別・フィラー除去済みテキスト）を受信して表示
```

設計上の決定事項：
- 署名付きURLの生成はサーバー側のみで行う（service_roleキーはクライアントに絶対に出さない）
- 音声ファイルはサーバーに保存しない。文字起こし完了後に即削除
- 削除失敗に備え、Supabase側のライフサイクル（後述のクリーンアップ）を保険として用意する

### A-2. Supabaseバケット設計

- バケット名：`audio-uploads`
- 公開設定：**非公開（private）**
- ファイルサイズ上限：200MB（バケット設定で制限）
- 許可MIMEタイプ：`audio/mpeg, audio/mp4, audio/wav, audio/x-wav, audio/m4a, audio/x-m4a, video/mp4, audio/webm, video/webm`
  - webm系はステージBの録音機能（Android Chrome出力）で必要になるため今のうちに含める
- ファイルパス規則：`{uuid}/{元ファイル名}` 形式。uuidはクライアントで生成（crypto.randomUUID()）し、衝突と推測を防ぐ
- Storageポリシー（RLS）：anonロールに対しINSERTのみ許可。SELECT/UPDATE/DELETEは不許可（読み出しはサーバーの署名付きURL経由、削除はservice_role）
- 注意：現状ログイン機能がないためanonキーでのアップロードを許容する。乱用対策（レート制限・使用量上限）はフェーズ3の使用量管理とセットで実装予定であることをコードコメントに明記しておく

クリーンアップ保険：Supabase無料プランではStorageの自動有効期限設定がないため、サーバー側に「アップロードから24時間以上経過したファイルを削除する」処理を用意する。実装はVercel Cron（vercel.jsonでスケジュール定義、1日1回）＋削除用APIエンドポイントとする。Cron設定が環境的に困難な場合は、文字起こしAPI呼び出し時に古いファイルを併せて掃除する方式（piggyback方式）にフォールバックしてよい。どちらを採用したか報告すること。

### A-3. 環境変数

| 変数名 | 置き場所 | 用途 |
|---|---|---|
| VITE_SUPABASE_URL | client/.env | supabase-js初期化 |
| VITE_SUPABASE_ANON_KEY | client/.env | クライアント直接アップロード |
| SUPABASE_URL | server/.env | 署名付きURL生成・削除 |
| SUPABASE_SERVICE_ROLE_KEY | server/.env | 同上（秘匿必須） |

- .env.exampleを両ディレクトリに用意・更新すること
- .gitignoreに.envが含まれていることを確認すること
- Vercel本番環境への環境変数登録はゆうの操作が必要（チェックポイント②）。登録すべき変数名と値の取得場所を、Vercelダッシュボードの画面遷移込みで案内すること

### A-4. クライアント側の変更

- ファイル選択→アップロードのフローを、Supabase直接アップロードに差し替える
- アップロード進捗表示を追加（200MBファイルでは進捗なしは体験が悪い）。プログレスバー等の装飾は最小限、パーセント表示程度でよい
- 受け入れ形式チェックにwebmを追加（mp3/mp4/wav/m4a/webm）
- エラーハンドリング：アップロード失敗、署名URL生成失敗、文字起こし失敗をユーザーに区別がつく文言で表示

### A-5. サーバー側の変更

- ファイル受信エンドポイント（multipart処理）を廃止し、`{ filePath }` を受け取って処理するエンドポイントに置き換える
- @supabase/supabase-jsをserverに追加し、署名付きURL生成（expiresIn: 7200）→ AssemblyAIへaudio_url渡し → 完了後削除、の一連を実装
- **STT層の抽象化（必須）**：AssemblyAI呼び出しを直接エンドポイントに書かず、プロバイダー非依存のインターフェースに切り出す
  - 例：`transcribe({ audioUrl, language }) → { utterances: [{ speaker, text, startMs, endMs }], audioDurationSec }`
  - AssemblyAI実装はこのインターフェースの一実装（`providers/assemblyai.js`等）として配置。将来Soniox / Deepgram / AmiVoice実装を追加してもエンドポイント側を変更せずに済む構造にする
  - フィラー除去・日本語スペース除去などの後処理は、プロバイダー実装の外側（共通層）に置く。ベンダー乗り換え時に資産として持ち運ぶため
  - audioDurationSec（音声長）を必ず返すこと。フェーズ3の使用量管理の集計キーになる
- 既存のAssemblyAI呼び出し・フィラー除去・日本語スペース除去のロジック自体は変更しない（上記の構造への再配置のみ）
- /api/healthにSupabase接続確認を追加（supabaseConfigured: true/false）

### A-6. 注意書きの追加（既存合意済み文面）

画面下部に以下4項目をプレーンテキストで表示する。この4項目は文面確定済みのため確認不要：

1. 各話者が15秒以上話すと識別精度が上がります
2. 発言量が少ない話者は識別されにくい場合があります
3. 話者名は右のペンアイコンから編集できます
4. 本サービスはAI処理に外部APIを使用しています。無料枠を超えた場合、運営側に費用が発生します。長時間の録音は分割してご利用ください。

### A-7. ステージA完了条件

- ローカルで5MB超のファイルが文字起こしできる
- 文字起こし完了後にSupabaseバケットからファイルが消えている
- Vercel本番で同様に動作する（チェックポイント③）
- 注意書きが画面下部に表示されている
- git log上でステージAの変更が機能単位でコミットされている

## ステージB：録音機能（スマホ対応・確認あり方式）

### B-1. ユーザーフロー

```
[録音開始ボタン] → マイク許可 → 録音中画面（経過時間表示・停止ボタン）
→ [停止] → 確認画面（録音時間・ファイルサイズ表示）
→ 「端末に保存」「文字起こしする」「破棄」の3択
→ 「文字起こしする」選択時のみ、ステージAのアップロード経路に合流
```

- 録音停止後の自動アップロードは行わない（確認あり方式で合意済み）
- 「端末に保存」はBlobのダウンロード（aタグ+download属性）。ファイル名は `meetlog-YYYYMMDD-HHmm.{拡張子}`
- 「文字起こしする」と「端末に保存」は排他ではない。保存してから文字起こしもできること

### B-2. 技術仕様

- MediaRecorder APIを使用。mimeTypeはMediaRecorder.isTypeSupported()で判定し、`audio/webm`（Android/PC Chrome）→ `audio/mp4`（iOS Safari）の優先順で選択
- **timesliceを5秒に設定し、ondataavailableごとにチャンクをIndexedDBへ逐次保存する**（ブラウザクラッシュ・誤タブクローズ対策）
  - IndexedDBのスキーマ：recordingセッションID＋連番＋Blobチャンク＋タイムスタンプ
  - 停止時は全チャンクを順番通りに連結して1つのBlobにする
- **復元フロー**：アプリ起動時にIndexedDBを確認し、未処理の録音チャンクが残っていれば「前回の録音が残っています（録音時間の概算を表示）。復元しますか？」と提示。復元なら連結して確認画面へ、破棄ならIndexedDBをクリア
- IndexedDBのクリーンアップ：文字起こし送信完了・端末保存・明示的破棄のいずれかでチャンクを削除。それとは別に7日以上前の残骸は起動時に自動削除
- **Wake Lock API**：録音中はnavigator.wakeLock.request('screen')で画面消灯を防止。非対応ブラウザ（旧iOS等）ではWake Lock取得失敗を握りつぶし、後述の注意書きでカバー。visibilitychangeで復帰時に再取得すること
- マイク取得は getUserMedia({ audio: { echoCancellation: false, noiseSuppression: false } }) を起点とするが、会議室の収音品質はデバイス差が大きいため、実装時にオプションの妥当性を再検討してよい（変更した場合は理由を報告）
- 長時間録音対策：経過時間が90分を超えたら録音中画面に「長時間の録音は精度・コストの面で分割を推奨します」の注意を出す（録音は止めない）

### B-3. 録音機能に伴う注意書きの追加（要承認）

以下の文面案をゆうに提示し、**承認を得てから**画面下部の注意書きに追加すること（文言は調整される可能性がある）：

- 案1：録音中は画面を閉じたり他のアプリに切り替えたりしないでください
- 案2：録音データの保存はご自身の端末で行えます。サーバーには保存されません

今後も実装中に新たな注意が必要だと判断した場合は、文面案を作成してゆうの承認を得たうえで随時追加してよい（最終的に削る前提で多めに出す方針）。

### B-4. ステージB完了条件

- PC Chromeで録音→確認→文字起こしが通る
- スマホ（Android Chrome実機）で録音→画面ロックされないこと→確認→文字起こしが通る（チェックポイント④）
- 録音中にタブを閉じて再訪問→復元ダイアログ→復元して文字起こしが通る
- 「端末に保存」でファイルがダウンロードでき、それを手動アップロードしても文字起こしできる（webm受け入れの確認を兼ねる）

## 並行タスク：STTベンダー実測比較（ステージA完了後、ステージBと並行可）

目的：AssemblyAIを継続するか乗り換えるかを、公称値ではなく実測で決める。結果はフェーズ3（リアルタイム対応・使用量管理・ベンダー確定）の設計入力となる。

- 比較対象：AssemblyAI（現行・基準値）／Soniox／Deepgram Nova-3／AmiVoice API
- 検証方法：ローカルで動く比較スクリプト（Node.js）を作成し、同一の実会議録音（複数話者・短い発話を含むもの）を各社のバッチAPIに投げて結果を並べて出力する。本体アプリには組み込まない
- 評価軸：(1) 話者識別の正答率（特に発言量が少ない話者）(2) 日本語の認識精度（固有名詞・数字）(3) 処理時間 (4) 実コスト（レスポンスの音声長×単価で算出）
- 無料枠の利用：AmiVoice＝毎月60分無料、Deepgram＝$200クレジット、Soniox＝登録時無料枠。新規アカウント登録とAPIキー取得はゆうの操作（チェックポイント⑥）
- 注意：テストに使う会議録音は実在の会議のため、ログ保存ポリシーに留意（AmiVoiceは「ログ保存なし」を選択）。比較スクリプトのAPIキーは.envで管理しリポジトリに含めない
- 成果物：比較結果のサマリー（表形式のmd）をゆうに報告。乗り換え判断はゆうが行う

## 実装順序とコミット計画（目安）

1. 既存コードレビューと問題点報告（修正方針の提示、軽微なリファクタリング）→ commit
2. STT層の抽象化（AssemblyAIをプロバイダー実装として再配置、後処理を共通層へ）→ commit
3. Supabaseクライアント導入・直接アップロード実装 → commit
4. サーバー側：署名URL生成・audio_url方式への切替・完了後削除 → commit
5. クリーンアップ処理（Cron or piggyback）→ commit
6. 注意書きUI（確定済み4項目）→ commit → 🆕報告・実装済み機能一覧の更新【ステージA完了】
7. （並行）ベンダー比較スクリプト作成・実行・結果報告
8. IndexedDB逐次保存つき録音モジュール → commit
9. 確認画面（保存/文字起こし/破棄）と復元フロー → commit
10. Wake Lock・長時間警告・注意書き追加（要承認）→ commit → 🆕報告・実装済み機能一覧の更新【ステージB完了】

各ステップで問題を発見した場合は、修正とリファクタリングを同一コミットに混ぜず、分けてコミットすること。
